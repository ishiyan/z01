# Blog Posts and Essays

Rules for articles, guides, tutorials, newsletter issues, and opinion pieces.

## Core Rules

1. **Lead with the concrete thing:** artifact, example, output, anecdote, number, screenshot, or code.
2. **Explain after the example, not before.**
3. **Keep sentences tight** unless the voice is intentionally expansive.
4. **Use proof instead of adjectives.**
5. **Never invent facts, credibility, or customer evidence.**

## Banned Patterns

Delete and rewrite:
- "In today's rapidly evolving landscape"
- "game-changer", "cutting-edge", "revolutionary"
- "here's why this matters" as a standalone bridge
- Fake vulnerability arcs
- Closing questions added only to juice engagement
- Biography padding that doesn't move the argument
- Generic AI throat-clearing that delays the point

## By Genre

### Technical Guides
- Open with what the reader gets
- Use code, commands, screenshots in major sections
- End with actionable takeaways, not soft recap

### Essays / Opinion
- Start with tension, contradiction, or specific observation
- One argument thread per section
- Opinions must answer to evidence

### Newsletters
- First screen does real work
- No diary filler up front
- Section labels only when they improve scannability

## Quality Gate

Before delivering:
- [ ] Factual claims backed by provided sources
- [ ] Generic AI transitions are gone
- [ ] Every section adds something new
- [ ] Formatting matches the intended medium
- [ ] Opening leads with a concrete thing, not preamble
