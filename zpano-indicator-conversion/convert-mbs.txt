  Do not copy tests as is, prefer our establishedtesting patterns in Go and Typescript.
  Tests such as "ToStringRest()", "SerializeTo", "SerializeFrom()" and "SerializationTest()" should be dropped.

In mbst-to-convert/john-ehlers/roofing-filter/ folder I have a mbst-indicator RoofingFilter.cs (ROOF) to convert.

This indicators implements three flavours of the roofing filter indicator described in a John Ehlers book.
The 6 pages of the book where it is described (along with three easy language listings) are in rf01.png ... rf06.png files.
If you can parse and read them, otherwise just skip.

There are three code listings in these pages:
- Listing 7‑1 (HP‑LP Roofing Filter): one‑pole high‑pass followed by the SuperSmoother.
- Listing 7‑2 (Zero‑Mean Roofing Filter): one‑pole high‑pass → SuperSmoother → another one‑pole high‑pass to remove the residual low‑frequency bias.
- Listing 7‑3 (Roofing Filter Indicator): two‑pole high‑pass followed by the SuperSmoother (this is the “better formulation” Ehlers describes and is inherently near zero‑mean).

The indicator has two parameters (hasTwoPoleHighpassFilter and hasZeroMean), which select one of three filter implementations:
- hasTwoPoleHighpassFilter = false and hasZeroMean = false
  Listing 7‑1 (HP‑LP Roofing Filter) which is One‑pole HP + SuperSmoother
  The Update1Pole() without the zero‑mean branch computes:
    HPᵢ = (1 − α/2)(xᵢ − xᵢ₋₁) + (1 − α)HPᵢ₋₁
    SSᵢ = c1(HPᵢ + HPᵢ₋₁)/2 + c2 SSᵢ₋₁ + c3 SSᵢ₋₂
  This matches Listing 7‑1, where α is based on the longest period and c1,c2,c3 are the SuperSmoother coefficients using the shortest period.
- hasTwoPoleHighpassFilter = false and hasZeroMean = true
  Listing 7‑2 (Zero‑Mean Roofing Filter) which implements zero‑mean version
  The Update1Pole() with hasZeroMean = true applies the same one‑pole HP again to the SuperSmoother output:
    ZMᵢ = (1 − α/2)(SSᵢ − SSᵢ₋₁) + (1 − α)ZMᵢ₋₁
  That reproduces Listing 7‑2 (HP → SS → HP), which reduces residual low‑frequency components to achieve a nominally zero‑mean output.
- hasTwoPoleHighpassFilter = true
  The hasZeroMean flag is ignored in the two‑pole path, so (true, true) behaves the same as (true, false).
  Listing 7‑3 (Roofing Filter Indicator)
  The Update2Pole() computes the two‑pole HP (the “.707”/√2 scaling appears in your coefficients via cos(π√2/Λ)) and then the same SuperSmoother.
  This is the formulation Ehlers prefers and is what Listing 7‑3 implements before the plotting/signaling logic.
  In the implementation, hasZeroMean is not used when hasTwoPoleHighpassFilter is true—consistent with the idea that the two‑pole HP already removes the longer‑cycle bias.

This MBST doesn't have unit tests yet, but in the same folder I have Julia implementation of this indicator roofing_filter_all.jl, its unit tests roofing_filter_all_tests.jl
The tests load the three CSV files, but you should extract input/expected data and inlibe them in our new unit tests.
The CSV files have the same input data in the first column, 'x'. These CSV files are:
- test_7-1_HP_LP_Roofing_Filter.csv
  use the second column. 'HP_LP_Roofing_Filter', as expected values for combination:
  hasTwoPoleHighpassFilter = false and hasZeroMean = false
  (Listing 7‑1, HP‑LP Roofing Filter)
- test_7-2_Zero_Mean_Roofing_Filter.csv
  use the third column, 'Filt2', as expected values for combination :
  hasTwoPoleHighpassFilter = false and hasZeroMean = true
  (Listing 7‑2, Zero‑Mean Roofing Filter)
- test_7-3_Roofing_Filter_Indicator.csv
  use the second column. 'Filt', as expected values for combination:

  (Listing 7‑3, Roofing Filter Indicator)

One of the supposed usecases of this "indicators" package is in chart indicators study.
A chartung app makes a list of indicators typically selected by user.
We need to categorize/classificate an indicator in several dimensions:
- by "minimal input data type".
  If charting app will going to feed Quote or Scalar data, it is better not to select an indicator which uses OHLC prices.
- by "uses volume" boolean.
  Some charting apps may have Bars without volume and just set 0 or NaN as a volume value.
  In this case it is useful to know if an indicator uses volume in calculations.
- by "output data type".
  If charting app wants to show a list of indicators with a Band output type in one of indicator's outputs
- by "role". This is categorized per individual output.
  If charting app wants to show a list of "smoothers" or "oscillators", or all indicator outputs that can be shown on price pane (i.e. may be visually compared with price).
  Based on indicators we actually have I can think of roles which don't fit into "smoother/oscillator" paradigm:
  - heatmaps, which usually displayed in a separate chart pane
  - cycle periods
  - fractal dimens
  - "polyline", e.g ZigZags and Fibbonacci levels
  Which "roles" can you suggest?
  Are there any other types in DFSP/technical analysis apart of "smoothers" and "oiscillators".

Let's brainstorm about this classification.

Also, speaking of indicator types, I realized that in core/ folder we have the type.go where we enumeration available indicators.
This is actually an identifier, not a type. When I hear "type", I think of "smoothers" and "oiscillators".
What do you think?
